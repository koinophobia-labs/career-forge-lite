# Founding-User Pilot Protocol

Internal tests established mechanical readiness. They cannot establish willingness to pay.
This protocol defines the controlled pilot that must run — with commerce still **off by
default** — before any of the proposed $49 / $79 / $99 prices may be described as
"supported."

## Scope

- **Participants:** the first 5–10 founding users, recruited individually.
- **Consent:** each participant explicitly agrees, in writing, to (a) the pilot protocol,
  (b) the counts-and-timings summary export, and (c) optional recruiter review of their
  materials. No consent → no data collection of any kind.
- **Founder review:** every pilot session's outputs are reviewed by the founder before any
  conclusion is recorded.
- **Commerce:** remains off. Nothing in the pilot asks a participant to pay. The $149
  human-reviewed service (a separate offer, see `/reviewed-service`) may be mentioned but
  never bundled into pilot participation.

## What is measured — never résumé content

Participants export the **pilot summary** from Settings → "Founding-user pilot summary."
It contains timings and counts only (schema `career-forge-pilot-summary-v1`, generated by
`src/lib/pilot-metrics.ts`, guarded by a fail-closed content check). Analytics events
remain event-name-only. Résumé content, claims, names, and employers are never collected.

Per participant, record:

| Metric | Source |
| --- | --- |
| Time to first approved evidence | pilot summary (`minutesFromStartToFirstApprovedEvidence`) |
| Time to usable export | pilot summary (`minutesFromStartToFirstExport`) |
| Export success (yes/no + blockers hit) | pilot summary counts + participant interview |
| Editing minutes per artifact | participant self-report during session |
| Artifact disposition: used / lightly edited / heavily edited / abandoned | participant interview, per artifact |
| Wrong-category items caught by review | pilot summary (`integrity.wrongCategoryItemsCaught`) + interview |
| Usefulness rating (1–5) | participant interview |
| Refund request (hypothetical: "would you have asked for your money back?") | participant interview |
| Used in a real application (yes/no, which artifact) | follow-up after 2 weeks |

## Blinded recruiter-review packet

For each consenting participant, build a packet with `node scripts/build-recruiter-packet.mjs`
containing three documents with **randomized labels** (the key is written to a separate
file the reviewer never sees):

1. The participant's previous résumé.
2. The Career Forge output (light participant edits allowed; log editing minutes).
3. A generic-AI baseline (same history pasted into a general-purpose chatbot with a plain
   "write my résumé" prompt).

Each recruiter scores every document 1–5 on:

- Credibility
- Clarity
- Target-role alignment
- Factual defensibility
- Likelihood of interview
- Editing burden (5 = ready to send)

Recruiters must not know which document is which, and must not be affiliated with the
project. Minimum: 2 independent recruiters per packet.

## Minimum human evidence before a price is "supported"

A price point ($49 Career Reset, $79 Job Search, $99 Career Switch) may be described as
**supported** only when ALL of the following hold:

1. **≥ 5 pilot participants** completed the tier's core workflow end to end.
2. **≥ 4 of 5** rated usefulness ≥ 4/5 for the artifacts that tier sells.
3. **Median editing burden** ≤ 15 minutes per artifact ("light edit"), self-reported.
4. **0 factual-defensibility failures** in recruiter review attributable to generation
   (participant-authored edits excluded).
5. Career Forge output **beats or ties the generic-AI baseline** on credibility and
   factual defensibility in blinded recruiter scoring, per participant.
6. **≥ 3 participants** used a pilot artifact in a real application.
7. **≥ 2 participants** independently state they would pay the proposed price
   (asked open-endedly, price named only after their answer).
8. The would-refund rate among completers is **0**.

Until every criterion is met, the correct public language remains: "proposed",
"hypothesis", "not validated". Internal test results — including everything in this
repository's test suites — can never satisfy these criteria on their own.

## Records

Pilot session records live outside this repository (they contain personal data). This
repository stores only: this protocol, the packet generator, the pilot-summary schema and
its content guard, and aggregate conclusions with no personal data.
